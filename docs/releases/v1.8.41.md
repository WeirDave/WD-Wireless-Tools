Both Sync arrows in Cloud Manager's Project Files toolbar now do the natural thing on orphan rows — **Sync ←** bulk-uploads local `.esx` files to Ekahau Cloud, and **Sync →** bulk-downloads cloud projects as `.esx` files. That last one required reverse-engineering Ekahau's client-side download worker (they build the ZIP in the browser, no server endpoint), and the resulting archives are byte-identical to what Ekahau AI Pro produces.

Suite `v1.8.40 → v1.8.41`. WD Cloud Manager `v3.43 → v3.45`.

## What changed

### 1. Sync ← extends to bulk upload (was queued as v1.8.40)

Previously **Sync ←** ("push local → cloud") only worked on matched pairs — renaming the cloud project to match the local file's name. Selecting 14 local-only `.esx` files sitting in Project Files greyed the button out, and the only recourse was clicking the per-row `←` upload arrow one file at a time.

Now, on the Project Files tab, **Sync ←** enables whenever the selection contains **either** matched pairs (existing rename behavior) **or** local-only `.esx` files (new — uploads via `upload_project`, the same 3-step presigned S3 flow the per-row arrow uses). Mixed selections do both in one confirm dialog.

### 2. Sync → extends to bulk download

Ekahau's web app doesn't expose a server-side "download this `.esx`" endpoint. Their web UI hits `GET /projectapi/v1/projects/{id}/batch` for the full project JSON, then fetches each floor plan image via a 302-redirect endpoint that hands out presigned S3 URLs, and finally assembles a ZIP archive **client-side in a Web Worker** (`project-download-worker-u2c06LO4.js`).

Reimplemented that whole flow in Python. `EkahauAPI.download_project(project_id)` now:

1. Calls `GET /projectapi/v1/projects/{id}/batch` (no `poTypes` query param → returns every entity type in one payload)
2. Iterates `images[]` from the response; for each image ID, calls `GET /projectapi/v1/projects/{id}/imageFiles/{imageId}?redirect=true` — the `requests` library follows the 302 automatically and returns raw image bytes from S3
3. Assembles a ZIP with:
   - `version` file containing `"2.0"`
   - One `{key}.json` file per top-level batch key, shape `{"key": value}` — e.g. `project.json`, `buildings.json`, `floorPlans.json`, `accessPoints.json`, `wallSegments.json`, `simulatedRadios.json`, etc.
   - One `image-{imageId}` entry per floor plan (no extension, raw bytes)

Verified against a real `.esx` downloaded via Ekahau's UI — 26 entries, all JSON contents match by semantic equality, all image bytes match byte-for-byte, `version` file matches. Ekahau AI Pro opens ours identically to their own downloads.

Each downloaded project lands in a folder named after the project itself, under your local `output_dir` — one folder per project, same convention Ekahau AI Pro uses. If you want to consolidate multiple projects under one site later, use **Move to site…**.

### 3. Both Sync buttons now visually enabled based on selection

The always-visible-but-disabled-when-inapplicable pattern from v1.8.39 now reflects the extended semantics:

| Selection on Project Files tab | Sync → enabled | Sync ← enabled |
| --- | --- | --- |
| matched pairs | ✓ rename local | ✓ rename cloud |
| local-only `.esx` files | disabled | ✓ upload to cloud |
| cloud-only projects | ✓ download to `.esx` | disabled |
| mixed | ✓ (all applicable actions run) | ✓ (all applicable actions run) |

Tooltips updated per-tab so the extended semantics are discoverable on hover.
