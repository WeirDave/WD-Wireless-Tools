Bulk-upload local `.esx` files to Ekahau Cloud. Cloud Manager's **Sync ←** button now does the natural thing when your selection includes local-only files — push them to the cloud in one action instead of clicking each row's individual upload arrow.

Suite `v1.8.39 → v1.8.40`. WD Cloud Manager `v3.43 → v3.44`.

## What changed

### 1. Sync ← extends to bulk upload

Previously **Sync ←** ("push local → cloud") only worked on matched pairs — renaming the cloud project to match the local file's name. If you had 14 local-only `.esx` files sitting in Project Files with no cloud counterpart, the button was greyed out and your only option was clicking the per-row `←` upload arrow one file at a time.

Now, on the Project Files tab, **Sync ←** enables whenever the selection contains either:

- matched pairs → renames the cloud project (existing behavior), or
- local-only `.esx` files → uploads them to Ekahau Cloud via `upload_project`

Both cases run in the same click, so a mixed selection (a few pairs plus some orphan uploads) does everything in one go. The button's tooltip explains the extended semantics; the confirm dialog reports the exact upload count (`Upload 14 local .esx files to Ekahau Cloud?`) so you can't fire it by accident.

### 2. Bulk uploads reuse the same backend

`upload_project` in `tools/cloud_manager.py` already polls Ekahau's project-listing API after each upload until the new project shows up — that's the same code path the per-row `←` button has been using. Bulk just loops over the selection sequentially, so from Ekahau's side each file goes through the identical validated 3-step presigned S3 flow. No new backend surface area, no parallel-upload race conditions.
