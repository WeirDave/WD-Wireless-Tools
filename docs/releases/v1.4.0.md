# WD Wireless Tools goes live on the web

Quick Walls now runs directly in a browser at **[weirdave.github.io/WD-Wireless-Tools/walls](https://weirdave.github.io/WD-Wireless-Tools/walls/)** — no install, no Python, drop an `.esx` and edit walls in about 5 seconds. The landing page at the same URL doubles as the front door for the whole suite, with download cards for Cloud Manager and Squirrel (which still need the desktop install).

## What changed

### 1. Hosted Quick Walls on GitHub Pages

Quick Walls was already 99% client-side — `.esx` parsing runs in the browser via JSZip, wall-type editing and keybind assignment are all local, and saving produces an `.esx` download. The one server dependency was template persistence (`/api/templates/*`).

`web/assets/js/walls.js` now has a runtime `HOSTED` flag:

```js
const HOSTED = typeof window !== 'undefined' && !!window.WD_HOSTED;

async function tplApi(action, data = {}) {
  if (HOSTED) return _hostedTplApi(action, data);
  const r = await fetch(`/api/templates/${action}`, { ... });
  return r.json();
}
```

When hosted, `_hostedTplApi` handles `scan` / `defaults` / `save` / `delete` / `get_folder` against `localStorage` instead. On first visit, it seeds `localStorage` with the "Recommended by WD" template so users don't land on an empty state. The Ekahau factory defaults are fetched from `../templates/ekahau_defaults.json` — a static JSON file the Pages build bundles alongside the site.

Desktop Quick Walls is byte-identical to before. `HOSTED` is only true when the Pages HTML injects `<script>window.WD_HOSTED = true;</script>`, which it does at build time.

### 2. Landing page + tool cards

Visiting the Pages URL shows a hero landing with three cards. Quick Walls is tagged with a pulsing green **Live in browser** pill and links to `./walls/`. Cloud Manager and Squirrel wear an amber **Desktop** pill and CTAs read "Download the suite →" pointing at the Releases page.

The two desktop tools have their own stub pages at `./cloud/` and `./organizer/` so any external link to those paths lands on a friendly "here's what this tool does, grab the download" page rather than a 404.

### 3. Pages build workflow

`.github/workflows/pages.yml` runs on every push to `main` that touches `web/`, `templates/`, or the workflow itself:

1. Copy `web/assets/`, `web/guide*.html`, and `templates/*.json` into a fresh `docs-build/`
2. Take `web/walls.html`, inject the `WD_HOSTED` flag right after `<head>`, and rewrite absolute `/assets/` paths to relative `../assets/` so the site works under the `/WD-Wireless-Tools/` project-pages prefix
3. Drop `hosted-index.html`, `hosted-cloud-stub.html`, and `hosted-organizer-stub.html` at the right paths
4. Upload as a Pages artifact and deploy via `actions/deploy-pages@v4`

The `web/` folder stays the single source of truth for both the desktop app and the hosted site. Zero drift, no duplicated files committed to the repo.

### 4. README badges + Live Demo callout

The README now leads with a green **Live Demo · Quick Walls** badge that links to the hosted URL, followed by a callout line encouraging visitors to try it before downloading.

## User-visible behavior change

- **Anyone on the internet can now use Quick Walls** by clicking [this link](https://weirdave.github.io/WD-Wireless-Tools/walls/). No login, no install, no server. Their `.esx` never leaves their machine — JSZip parses it locally.
- **Hosted templates live in `localStorage`** and don't sync between browsers or devices. If someone opens the hosted Quick Walls on their laptop, saves a template, then opens it on their phone, the template won't be there. Desktop templates (stored on disk in the templates folder) are unaffected.
- **The `docs/` folder in the repo now holds release notes only** (this file lives at `docs/releases/v1.4.0.md`). It is NOT the Pages source — that comes from a fresh `docs-build/` created in CI.
