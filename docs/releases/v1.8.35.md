Uploading a local `.esx` to Ekahau Cloud now shows the new project in the list immediately — no manual refresh, no waiting for the polling timer.

Suite `v1.8.34 → v1.8.35`. WD Cloud Manager `v3.39 → v3.40`.

## The bug

After uploading a local `.esx` to Ekahau Cloud, the freshly-uploaded project sometimes didn't appear in the Cloud Manager row list. Users had to either wait for the polling timer or hit browser refresh manually.

## Root cause

Ekahau's project-listing API has a **1–2 second propagation delay** (occasionally longer) between the upload endpoint returning success and the new project actually appearing in `/site-management-api/v1/datasetListing`. The frontend called `refreshData()` immediately after upload — before Ekahau's own listing had caught up — so the new project was still missing from the response.

Only the auto-assign-to-site code path handled this (a hard-coded `time.sleep(1.5)`); plain uploads got no buffer.

## Fix

`CloudManager.upload_project()` now **polls the project listing every 500 ms up to 6 s**, waiting for the new project to actually appear before returning. The returned `datasetId` is confirmed present, so a subsequent frontend refresh is guaranteed to see the project.

```python
before_ids = {p["id"] for p in self.api.get_projects()}
result = self.api.upload_project(esx_path)
new_id = None
for _ in range(12):        # 12 * 0.5s = 6s max
    time.sleep(0.5)
    after = self.api.get_projects()
    candidates = [p["id"] for p in after if p["id"] not in before_ids]
    if candidates:
        new_id = candidates[0]
        break
```

If propagation takes longer than 6 s (rare), the response includes a `warning` field so the frontend shows an amber toast — the next polling refresh will still catch it.

Frontend `uploadFromLocal()` reverted to a single `refreshData()` call — no more timers or races between JS and Ekahau's eventual consistency.
