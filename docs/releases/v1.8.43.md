Quick Walls now has a **🖼 Visual Wall Swap** mode — open your `.esx`, click the button, and you get the actual floor plan on a canvas with every wall drawn on top in its wall-type color. Marquee a box (or click walls one at a time), pick the wall type you want them to be, click Swap. Selected walls can also be **deleted outright** — useful for cleaning up the artifact walls that Ekahau's auto-detection sometimes creates (long spurious segments extending off steel columns, disconnected stubs, etc.). For the "just change every Light Window Glass in the whole project to Thick Window Glass" case, there's a **Quick swap by type** shortcut that skips the drawing entirely.

Suite `v1.8.42 → v1.8.43`. WD Quick Walls `v7.18 → v7.26`.

## What changed

### The gap this fills

Ekahau doesn't have a "select every wall of type X and change it to type Y" bulk action. You can marquee-select walls in Ekahau AI Pro, but only one floor at a time and only walls you can visually see — no way to say "everything of this type, everywhere." So if you drew the whole building with **Light Window Glass** and later realized it should have been **Thick Window Glass**, your options were: redraw, or click each wall one at a time.

Now:

- Open the `.esx` in Quick Walls
- Click **🖼 Visual Swap** (new button, next to **+ Add Wall Type**)
- Full-screen modal opens with the floor plan rendered
- Right sidebar shows every wall type on this floor with its count
- Drag a marquee → pick a target type → Swap
- Or: **Quick swap by type** — From `Light Window Glass` → To `Thick Window Glass` → Scope: `Whole project` → Swap all

### Modal anatomy

**Topbar** — WD title, floor picker (dropdown of every floor in the project, with wall count per floor), running status (`128 walls on this floor · 1191 total in project`), **Save .esx** (primary blue — writes the whole ZIP without leaving the modal, so you can incrementally save between batches of edits), Close.

**Left toolrail** (48px vertical strip):
- ▢ Marquee — drag a rectangle to select every wall it touches
- ◉ Click — toggle individual walls (shift-click to add without clearing)
- ✋ Pan — drag to move the view (or hold **Space** with any tool)
- Zoom to fit / Zoom in / Zoom out
- Clear selection

**Canvas** — the actual floor plan image, rendered at full resolution with a subtle darken tint so wall colors read cleanly. Walls draw as colored polylines using each type's own color from the palette. Selected walls get a fat white halo + colored stroke on top so they stand out from adjacent walls of the same type. Wheel zooms toward the cursor.

**Right sidebar** — three panels:
- **Wall types on this floor** — every type with a color swatch, name, and count. **Click any row to select every wall of that type on the current floor** — same behavior as marquee-selecting them all.
- **Selected** — running count, broken down by source type (`23× Light Window Glass · 24× Concrete`), plus **Change to** dropdown and **Swap** button, and a red **Delete selected** button for removing bogus walls entirely.
- **Quick swap by type** — no drawing required. From / To / Scope (`This floor only` or `Whole project`) → **Swap all**.

**Keyboard**: `M` marquee, `V` click, `H` pan, `Space` (hold) temporary pan, `Esc` clear selection (or close modal if nothing selected), `Ctrl+A` select all on this floor.

### Data flow (why saving Just Works)

The modal reads `wallSegments.json`, `wallPoints.json`, and `floorPlans.json` out of the same JSZip that Quick Walls already loaded. Swaps mutate the segment objects in memory, then write the updated `wallSegments.json` back into the JSZip immediately. When you close the modal and hit **Save the *.esx** in Quick Walls, the modified segments are packaged into the output ZIP alongside your wall-type edits. **One save, both changes.**

Nothing is committed until you Save — closing the modal without Save keeps changes staged but not written to disk.

### Rendering & geometry

The tricky part was that Ekahau doesn't store wall coordinates on the segment itself. Each `wallSegment` references two `wallPoint` IDs; each `wallPoint` carries the `{x, y, floorPlanId}` inside its `location`. The renderer:

1. Loads `wallPoints.json` into a `pointId → {x, y, floorPlanId}` map
2. For each `wallSegment`, walks the id references to resolve endpoints and derive the segment's floor
3. Falls back to inline geometry if a hand-crafted or older export uses that shape instead

Coordinate space is the floor plan image's native pixel space (which is also how Ekahau stores its points), so walls line up on the drawing without any scale math.

Marquee hit-testing uses proper line-vs-rectangle intersection (any-endpoint-inside + line-crosses-any-edge), so a wall that passes through the marquee gets picked up even when both endpoints are outside the box. Click-to-select uses point-to-line-segment distance with a 6px tolerance in screen space (so it scales with zoom — you don't have to hit the pixel).

### Version bumps

Quick Walls jumped `v7.18 → v7.23`:
- `v7.19` — added the modal shell and button
- `v7.20` — fixed the geometry loader to walk through `wallPoints.json` (initial version was looking for inline endpoints and a `floorPlanId` on the segment — both wrong for the real schema, which is why the first paint showed "0 walls" on a floor with 1191)
- `v7.21` — read `location.coord.{x,y}` instead of `location.{x,y}` (verified against real .esx), and render in the floor plan's declared coord space (`floorPlans.json` width/height, e.g. 10000×7500) instead of image pixel space — so walls actually land where they were drawn
- `v7.22` — line thickness now grows with zoom (`sqrt(zoomRatio)` curve, baseline 3px at fit, capped at 14px) — old constant 3px looked like hairlines when zoomed in. Pan-tool icon swapped to a proper hand.
- `v7.23` — **Delete selected walls** button in the Selected panel (red destructive style, confirms before wiping). Removes selected segments from `wallSegments.json`; leaves un-referenced `wallPoints` entries alone (Ekahau ignores orphans; skipping the point-pruning avoids any chance of dropping a point still shared with a neighboring wall).
- `v7.24` — **Save .esx** button in the modal topbar so you don't have to close-then-save every time you make a batch of edits.
- `v7.25` — Header restructured to match the rest of the WD tools suite: navy `#0e1b26` background, WD-blue title span, filename pill badge on the left. Floor picker + wall counts + zoom % moved to a narrow centered footer strip below the header (cleaner header, and the on-canvas zoom HUD is gone since it was redundant). Zoom % is now measured relative to zoom-to-fit — the whole plan reads as **100%** when it first loads (previously it showed the raw scale ratio, so a 10000-unit-wide floor plan opened at 16%, which was confusing).
- `v7.26` — Fixed the "hold space + wheel zoom + drag + release space" trap where the view snapped back to fit. Root cause: browsers treat spacebar as "activate the focused button". If the user had previously clicked (say) Zoom-to-fit, its focus persisted, so releasing space fired a synthetic click on it. Fixed with two-layer defense — `preventDefault` on space in both keydown and keyup, plus a delegated `blur()` on any toolbar button after click so focus never persists.

## Verification

3. Floor plan appears with walls drawn on top, colored by type
