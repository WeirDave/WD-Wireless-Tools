# Feature: "Not a match" — permanently un-pair mismatched rows

Cloud Manager's fuzzy matcher was pairing projects that share a site code but are clearly different (e.g. `SNAN2 - 3030 S Susan St, Santa Ana, CA 92704 - Cleanroom - PD.esx` ↔ `SNAN2 - 3030 - Remediation PD.esx`). The user had no way to say "these two are not the same project" — the pair kept showing up as a Mismatch on every refresh, dirtying the dashboard count and inviting an accidental sync.

v1.5.0 adds a **Not-a-Match** decision that persists across sessions.

## What's new

### ≠ button on Mismatch rows

Every mismatched row now has three gut buttons instead of two:

| ➡ | Apply cloud name to local |
| ⬅ | Apply local name to cloud |
| **≠** | **Not a match — never pair these two again** *(new)* |

Click ≠ → confirm dialog shows both names → both items drop out of the pair and reappear as orphans (Cloud Only + Local Only) on the next refresh.

### Persistent storage

Decisions are stored in `~/.wd_wireless_tools/not_matches.json` — the same per-user config dir that already holds `config.json` and `cookies.json`. No new "Data" folder is created; the existing pattern is reused. The file is a plain JSON list of `{cloudId, localPath, cloudName, localName, addedAt}` pairs.

Pair identity is `(cloudId, localPath)`:
- Cloud IDs are Ekahau UUIDs — stable across renames on the cloud side.
- Local paths are stable until the user moves or renames the file locally. If they do, the exclusion silently becomes moot (won't apply to the new path), which is the right behavior — a renamed file is a different file, and the matcher should be free to consider it again.

### Manage Not-a-Match modal

Menu → **Manage Not-a-Match** opens a modal listing every excluded pair with an **Un-mark** button on each row. Un-mark instantly lets the matcher consider that pair again on the next refresh. The modal also shows the on-disk path of `not_matches.json` for anyone who wants to inspect or back up the file.

## How the matcher uses it

`build_matches(cloud_items, local_items, excluded)` now takes an `excluded` set. Every one of the three passes (exact name, site code, fuzzy) skips a candidate whose `(cloud.id, local.path)` key is in the set. Result:

- The blocked cloud project falls through to **Cloud Only** (or picks up a different local match with weaker evidence, if one exists).
- The blocked local file falls through to **Local Only** (same caveat).
- No forced pairing — both items are honestly reported as unmatched.

The exclusion list is loaded fresh on every `get_data` call, so mark/unmark takes effect on the next refresh without a server restart.

## Verification

1. Restart the Flask server (backend gained new endpoints).
2. Hard-refresh the browser (`Ctrl+Shift+R`) so the new JS/CSS load.
3. On the Project Files tab, find a false-positive mismatch — click ≠ → confirm.
4. Row splits into two orphans (Cloud Only + Local Only) on the next refresh. Mismatches count in the dashboard drops by one.
5. Menu → **Manage Not-a-Match** → verify the pair is listed → click **Un-mark** → refresh → the pair returns to Mismatches.
6. Restart the server → the exclusion is still remembered (persisted in `~/.wd_wireless_tools/not_matches.json`).
