Six bug fixes coming out of a full code audit — two security issues, one broken hamburger menu, one silent-XSS-defeat helper, one theme-toggle label sync bug, and one duplicate-upload footgun.

Suite `v1.8.35 → v1.8.36`. WD Cloud Manager `v3.40 → v3.41`.

## Background

I ran four parallel Claude agents auditing (1) the Python backend, (2) the frontend JS, (3) the CSS, and (4) the HTML for consistency + accessibility. This release is the "fix now" tier — six items that were either exploitable, silently broken, or user-visibly wrong. The cleanup-tier items (dead CSS, stale README versions, accessibility) will land in v1.8.37.

## 1. XSS through user-controlled strings in inline `onclick` handlers

**Impact:** any Ekahau Cloud site/project name, local folder name, or `.esx` filename that an attacker could set could execute arbitrary JavaScript in the user's browser.

**Root cause:** `WD.escAttr` converts `'` to `&#39;` — safe for a pure HTML attribute value. But the pattern `onclick="fn('${escAttr(name)}')"` puts the value inside a **JS single-quoted string that lives inside a double-quoted HTML attribute**. The browser's HTML parser decodes `&#39;` back to `'` **before** the JS engine sees the attribute value — reopening the string literal. A project named `x');alert(1);('` executes `alert(1)`.

**Fix:** added `WD.escJsStr(s)` — backslash-escapes `\ ' < \n \r` (HTML preserves backslash-escapes as literals) then HTML-escapes only `&` and `"` for the surrounding attribute. Introduced `j()` and `pj()` local aliases in `cloud.js` and swapped every `'${a(x)}'` / `'${p(x)}'` pattern (53 sites) to `'${j(x)}'` / `'${pj(x)}'`. Also fixed `organizer.js` (used `esc()` — even worse — for its onchange handler) and `walls.js` (unescaped `.esx` colors interpolated into `style=""` attributes, plus `attenuationFactor`/`thickness` values into HTML text).

**Verified with a live exploit test** — before: `alert(1)` fires. After: the full evil string arrives at the handler as data, no execution.

## 2. Path traversal in `merge_execute` via client-supplied `rel`

**Impact:** a crafted request to `/api/cloud/merge_execute` with `ops=[{"rel":"../../secrets.txt","action":"move"}]` would `shutil.move` files outside the intended `src`/`dst` directories.

**Fix:** per-op `_assert_inside(s, src)` and `_assert_inside(d, dst)` guards. Traversal attempts get recorded in `errors[]` and the loop continues.

## 3. Path traversal in `rename_local` via `new_name`

**Impact:** a rename with `new_name = "../../../loose.esx"` would move the file outside `output_dir`.

**Fix:** reject `new_name` values containing `/`, `\`, or `..` segments outright; add a belt-and-suspenders `_assert_inside(new, old.parent)` before the rename runs.

## 4. Cloud Manager hamburger menu was broken

**Impact:** clicking Settings / Change Local Folder / Manage Not-a-Match / Refresh / About from the top-left hamburger did nothing.

**Root cause:** all five items chain `closeMainMenu(); action()` — but `closeMainMenu` was never defined anywhere in the codebase. The first `ReferenceError` killed the entire onclick handler, so `action()` never ran.

**Fix:** added a 3-line `closeMainMenu()` to `cloud.js` that removes the `open` class from `#mainMenu`.

## 5. Theme toggle label frozen on Home + hosted landing

**Impact:** on the two landing pages, clicking the theme pill flipped the icon (🌙 ↔ ☀️) but the "Dark" / "Light" label never updated.

**Root cause:** `home.html` and `pages/hosted-index.html` had `<span id="themeTxt">Dark</span>` — missing the `class="txt"` that `WD.syncThemeUI` queries via `querySelector('.txt')`. Every other page had it.

**Fix:** added `class="txt"` to both spans.

## 6. `assign_to_site` failure reported the whole upload as failed

**Impact:** if the upload succeeded but the follow-up "assign to site" call raised (transient 500, permission error, site deleted mid-upload), the outer `except` returned `{"error": ...}` — even though the `.esx` was already committed to cloud. User retried thinking the upload failed → duplicate cloud project. Ironic, given Duplicates detection is the tool's flagship feature.

**Fix:** wrapped `assign_to_site` in its own try/except. On failure the response now returns `{"ok": True, "uploaded": True, "datasetId": <id>, "assignError": <msg>, "warning": ...}` and the UI shows an amber toast instead of a red one.
