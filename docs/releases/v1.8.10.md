# v1.8.10 — Duplicates row actions fixed

Two bugs Dave hit in the Duplicates view:

1. **"View folder contents"** (👁) button on local cluster items errored out.
2. **Per-row delete** (🗑) didn't do anything — but the bulk "Delete checked" button worked.

Both are now fixed.

Suite `v1.8.9 → v1.8.10`. WD Cloud Manager `v3.19 → v3.20`.

## Bug 1 — "View folder contents" broken

The button called `openPeek(it.path)` which looks the path up in `data.matched` / `data.localOnly` via `localByPath()`. But in the Duplicates view, `data` holds the duplicates payload (clusters) — Sites/Projects data isn't loaded. `localByPath` returned `undefined` → "No file details available" toast.

Semantic mismatch anyway: cluster items are **files** (`.esx`), not **folders**. The adjacent 📁 "Show in Explorer/Finder" button already opens the containing folder, highlighting the file. The 👁 button was redundant and broken.

**Fix**: removed the 👁 button from local dup-items. Cloud items keep their 👁 "View site contents" button — that one uses `openCloudPeek` which does its own API call and doesn't depend on Sites/Projects data.

## Bug 2 — Per-row delete no-op

The onclick was rendered with `a(iid)` (attribute-escape only). For local items where `iid = it.path`, that's a Windows path with backslashes. Inside a JS-string attribute, `\U`, `\f`, `\N` etc. are interpreted as JS escape sequences — `C:\Users\dpaine\...` becomes `C:UsersdpaineFile.esx` when the string is parsed. `dupDeleteOne` then couldn't find the item in `cl.items` (path mismatch) and silently returned.

`dupDeleteChecked` didn't have the same bug because it reads `iid` from `row.dataset.iid` (data-attribute), which stores the string as-is — no JS-string escape pass.

**Two-part fix**:

- Onclick now emits `p(iid)` for local items (converts `\` → `/`) instead of `a(iid)`. Cloud items still use `a(iid)` because their IDs are UUIDs — no backslashes, no problem.
- `dupDeleteOne` normalizes both sides before comparing: `s => s.replace(/\\/g, '/').toLowerCase()`. So even if the passed path uses forward slashes and `cl.items[].path` uses backslashes (which it does on Windows), they match.
- Added a `toast('Could not locate that item', 'error')` fallback so a future path-format regression is loud instead of silent.

## Verified in browser

- Duplicates row action count: 2 for local (Show-in-Explorer, Delete) — was 3.
- Delete button onclick contains no backslashes; forward-slash path intact through the escape.
- `dupDeleteOne`'s find succeeds against the backend's backslash paths.
