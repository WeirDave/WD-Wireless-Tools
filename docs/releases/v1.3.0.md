# Duplicates — find the copies you didn't know you had

Cloud Manager now understands that "SYDN2 - Baseline Survey.esx" and "SYDN2 Baseline Survey.esx" are the same survey stored twice. A new **Duplicates tab** clusters these near-copies across your local folder AND your Ekahau Cloud, shows you which one is newest and largest, and gives you one-click cleanup. Rows in the Sites and Project Files tabs that belong to a duplicate cluster now carry a small **≈** badge — click it to jump straight to the cluster and decide.

## What changed

### 1. New Duplicates tab

A third tab sits next to Sites and Project Files. It scans every cloud project and every local `.esx` at once, groups them by a punctuation-normalized name (dashes, dots, extra spaces don't matter), and shows any group with two or more items as a cluster.

Each cluster card shows:

- The normalized name and a shape badge — **Mixed** (blue), **Local only** (purple), or **Cloud only** (cyan)
- Count pills — ☁ N cloud, 💻 N local
- A row per copy with: side icon, name, **size**, **modified date**, location (site name or folder name), a green **matched** pill if that copy is the one currently paired in the Sites view, and per-item actions (👁 peek, 📁 reveal in Explorer, 🗑 delete)
- **Newest** (green tint) and **Largest** (blue tint) markers automatically applied to the winners, so decision-making is visual — no need to squint at bytes
- Cluster-level bulk actions: **Keep newest — delete rest**, **Keep largest — delete rest**, or check individual copies and hit **Delete checked**

Sort order is deliberate: **Mixed clusters first** (usually the most actionable — they explain "why is this local file an orphan when I already have that survey?"), then local-only, then cloud-only, each block internally sorted by total cluster size descending.

The dashboard on this tab swaps its usual match/orphan cards for four duplicate-scoped filters: **Clusters** (total), **Mixed**, **Local Only**, **Cloud Only**. Click any card to filter.

### 2. ≈ badge on Sites and Project Files rows

Whenever you're viewing the Sites or Project Files tab, Cloud Manager quietly fetches the duplicates map in parallel. Any row whose file also appears in a duplicate cluster gets a small **≈** amber badge next to its name. Click the badge to jump to the Duplicates tab with that specific cluster expanded, scrolled into view, and pulse-highlighted.

This means you don't have to remember to check the Duplicates tab — the passing signal is always in front of you when you're browsing your normal views.

### 3. Local .esx metadata now includes modification time

`get_local_esx_files` in `tools/cloud_manager.py` used to return `{name, folder, size, path}`. It now also returns `mtime` (unix seconds) so the Duplicates tab can rank copies by "which one did I edit most recently" — the most common tiebreaker when you're cleaning up.

### 4. Release notes style switched to hand-authored narrative

`.github/workflows/release.yml` no longer relies on GitHub's auto-generated commit-list release notes. Instead, it reads from a per-release Markdown file at `docs/releases/vX.Y.Z.md` (the file you're reading is the first one). If the file is missing, the release still publishes with a blank body rather than failing.

This means every release from here on out carries the same narrative structure as the WaxFrame-Professional releases — "What changed", "User-visible behavior change", "Verification", "Files changed" — instead of dumping raw commit subjects.

## User-visible behavior change

- **New tab** — the tab bar in Cloud Manager now has three tabs. Existing Sites and Project Files behavior is unchanged.
- **≈ badges** — you'll see amber ≈ icons next to any row that has duplicates elsewhere. If you don't have any duplicates, no badges appear. Non-obtrusive by design.
- **Nothing gets deleted automatically.** Every deletion path in the Duplicates tab requires a confirmation dialog that lists every file it will remove, with side, name, and size. Cancel exits without touching anything.
- **The bulk "keep newest / keep largest" actions do not touch the "matched" copy specially.** If you had a matched local file that happens to be older than a stray copy, "keep newest" will delete the matched one. Check the matched pill before you click.
