# Cloud Manager v3.12 — Size + date matching, no more whack-a-mole

You hit a false match on SNAN2 3030 (cloud "Cleanroom - PD" was pairing with local "Baseline Survey" — 50× size difference, obviously not the same file) and found the ≠ button was working *technically* — the pair was excluded — but the matcher immediately paired the same cloud project to the *next* local file sharing the site code. Whack-a-mole.

Root cause: the code-match pass (Pass 2) only refused a pair when name-based discriminators (building token, street number) fired. If both files just shared "SNAN2 3030", they'd pair — regardless of survey phase, file size, or age. And the fuzzy-similarity threshold on Pass 3 didn't apply to Pass 2 at all, so any code-match with 0.15 similarity was fine.

Suite `v1.8.1 → v1.8.2`. WD Cloud Manager `v3.11 → v3.12`.

## New match criteria: **name, size, date**

Added a `pair_conflicts(cloud_item, local_item)` gate that runs before Pass 2 and Pass 3 pairing. A pair is rejected if ANY of three signals disagree:

- **Name discriminators** (existing) — building token, street number, plus new survey-phase tokens (baseline / predictive / remediation / cleanroom / post-install / pre-install / as-built / as-ran / TVR-validation). Convention-specific; opt-in per name.
- **Size** — both sides above the 500 KB floor AND max/min > 5×. Universal signal that works for generic names like "Building 1" where naming heuristics have nothing to grip. Same-project pairs stay within ~2× under normal edits/versioning; 5× is the "clearly different content" threshold.
- **Date** — both sides carry a valid mtime AND they're > 5 years apart. Backstop for small files that survive the size floor.

All three are floors, not tuners — they only fire on obviously-different files. Same-project cloud/local pairs with legitimate size drift or months-apart edits still match cleanly.

## Data plumbed

- `build_projects_data()` — cloud project dicts now carry `size` (int bytes from `pr["statistics"]["size"]`) and `mtime` (unix seconds via existing `_parse_cloud_mtime` helper). Local .esx dicts get the same fields (already had them internally). Both flow to the matcher.
- `build_matches()` — Pass 2 and Pass 3 now call `pair_conflicts(c, l)` instead of the older name-only `discriminators_conflict(c["name"], l["name"])`.
