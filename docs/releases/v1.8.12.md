# v1.8.12 — Duplicates: hide normal pairs, pair-safe cluster actions

Dave's realization once he understood what the view was doing:

> "The duplicate projects currently lists every project and then of course shows the matches, and then if I scroll down it's going to eventually show me ones that are not matched. But what we need to do is build in logic so that the matching ones don't show as long as there isn't a 3rd or 4th duplicate. …And the 'keep newest / keep largest' buttons — if these are matching, ones on the cloud side and ones on the local side, that's exactly what we want; those would be in symmetry."

Two changes flow from that:

1. **Filter out normal pairs.** 1-cloud + 1-local clusters where both sides are matched are just… pairs. They belong on the Projects tab, not here.
2. **Rework the cluster action buttons.** "Keep newest / Keep largest — delete rest" is dangerous on a Mixed cluster with a matched pair — the newest might be the cloud, so the button would delete both local files (including the matched one), breaking the pair. Replaced with **"Delete N extras (keep the pair)"** — a pair-preserving action.

Suite `v1.8.11 → v1.8.12`. WD Cloud Manager `v3.21 → v3.22`.

## Backend filter

`build_duplicates_data` now skips any cluster where `sides.cloud == 1 AND sides.local == 1 AND every item is matched`. Everything else falls through:

- 3+ items in the cluster → shown (real duplicate somewhere)
- Any unmatched item → shown (something isn't paired — worth surfacing)
- 1-on-each-side pair that failed to match (discriminator conflict, content drift) → shown (this is the "something's off" case)

Concrete impact on Dave's account: **103 clusters → 6**. All 6 are Mixed 3-item clusters (matched cloud + matched local + 1 extra local). Every one has actionable work: delete the extra, keep the pair.

## Shape-aware action buttons

The cluster's `.dup-actions-bar` now looks at `matchedCount` and `extraCount`:

| Cluster shape | Buttons shown |
| --- | --- |
| **Local-only** or **Cloud-only**, all items unmatched | `Keep newest — delete rest` · `Keep largest — delete rest` · `Delete checked` — every item is a candidate, the aggressive picks are safe. |
| **Mixed** with a matched pair AND extras (the common case now) | `🗑 Delete N extras (keep the pair)` (amber) · `Delete checked` — deletes only unmatched items via `dupDeleteExtras`. |
| **Mixed** with all items unmatched (rare — content drift or discriminator conflict) | `Delete checked` only — user must decide manually. |

New `dupDeleteExtras(key)` walks `cl.items`, filters to `!it.matched`, and passes to `_bulkDeleteItems`. Same confirm dialog as everything else — one click, one confirm, done.

## Explanation banner rewritten

> Clusters of files that share a normalized name AND have **at least one extra copy beyond the normal cloud↔local pair**. Normal 1-on-each-side pairs are hidden — they're already visible on the Projects tab. Within each cluster, faded items are already paired; **bold + amber** items are the extras worth deleting or merging.

Legend under the banner still shows the three shape tags, but the descriptions now match the new filter semantics.
