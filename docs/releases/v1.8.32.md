Firefox-compatible favicon swap. v1.8.31's HTML media queries broke the favicon in Firefox because Firefox ignores `media` on `<link rel="icon">` — Dave saw no favicon at all in Windows light mode. Fix moves the swap to JS via `window.matchMedia`, which works in every browser.

Suite `v1.8.31 → v1.8.32`. Behavior stays the same: white icon in browser dark mode, color icon in browser light mode.

## The Firefox bug

v1.8.31 shipped paired icon `<link>` tags — one per color scheme, gated by `media="(prefers-color-scheme: light|dark)"`. Chrome, Edge, and Safari respect the media queries and pick the right one. **Firefox ignores media on link[rel=icon] entirely** and picks the last icon in the DOM. Since the last link was `-white-32x32.png`, Firefox always used the white icon — invisible on light browser chrome.

## Fix

1. HTML rolled back to a single set of icon links per page (no `media`, no duplicates):
   ```html
   <link rel="icon" href=".../wd-wireless-tools-v8.0-multi-size.ico" sizes="any">
   <link rel="icon" type="image/png" sizes="32x32" href=".../wd-wireless-tools-v8.0-32x32.png">
   <link rel="apple-touch-icon" sizes="180x180" href=".../wd-wireless-tools-v8.0-180x180.png">
   ```
2. `WD.syncFavicon()` now reads `window.matchMedia('(prefers-color-scheme: dark)').matches` — the browser signal, not the app's `data-theme` — and rewrites each icon `href` to swap between color and white variants.
3. Element-replace (rather than plain `href` mutation) forces Firefox to re-fetch — some browsers cache favicons aggressively and won't refresh on a bare href change.
4. `WD.syncFavicon` is called on `DOMContentLoaded` and registered as a listener on the media-query change event, so switching Windows/macOS/browser between light and dark updates the tab icon live without a page reload.
5. The app's own theme-toggle button does **not** touch the favicon anymore. That was confusing intent and is now cleanly separated.
